\section{Introduction}

The current AI application stack is crowded. A production agentic system may include an application framework, an agent orchestration library, a model gateway, one or more provider APIs, tool protocols, durable workflow infrastructure, observability, and evaluation. OpenAI's Agents SDK, for example, exposes agents, tools, handoffs, guardrails, sessions, human-in-the-loop mechanisms, and tracing in Python and JavaScript/TypeScript \citep{openai_agents_python,openai_agents_js,openai_agents_tracing}. LangGraph emphasizes durable execution, streaming, persistence, and human-in-the-loop workflows \citep{langgraph_overview}. Cloudflare AI Gateway provides analytics, logging, caching, rate limiting, retries, and model fallback \citep{cloudflare_ai_gateway}. LiteLLM exposes a provider gateway abstraction and model routing strategies across many providers \citep{litellm_routing}. The Model Context Protocol (MCP) standardizes tool exposure through named tools with schemas and metadata \citep{mcp_tools}. OpenTelemetry has moved GenAI telemetry toward specialized semantic conventions for spans, metrics, and events \citep{otel_genai}.

This is progress. It is also fragmentation. Execution decisions are scattered across application code, framework internals, provider-specific APIs, gateway settings, workflow retries, evaluation jobs, and tracing backends. The same concerns recur: model choice, provider choice, context assembly, duplicate elimination, prompt-cache layout, retrieval ordering, side-effect control, validation, retry policy, human approval, trace capture, cost accounting, and data governance.

\migaki{} begins from a narrow claim: agentic systems need an execution layer that represents model calls, tool calls, retrieval, context artifacts, validators, side effects, and constraints as an optimizable execution graph. The goal is not to build another agent framework. The goal is to make AI execution portable, measurable, and optimizable.
